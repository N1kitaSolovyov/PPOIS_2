# Отчёт по лабораторной работе №2

## 1. Введение

**Цель работы:** изучение принципов построения графического пользовательского интерфейса и разработка программной системы с графическим интерфейсом на языке Python с использованием шаблона проектирования Model-View-Controller (MVC).

**Вариант задания:** 13

**Тематика предметной области:** учёт железнодорожных поездов (расписание). В соответствии с вариантом 13 таблица включает следующие атрибуты:
- Номер поезда
- Станция отправления
- Станция прибытия
- Дата и время отправления
- Дата и время прибытия
- Время в пути (вычисляется автоматически)

**Условия поиска и удаления** (из варианта):
- по номеру поезда или дате отправления;
- по времени отправления (задается верхний и нижний предел) или времени прибытия (задается верхний и нижний предел);
- по станции отправления или станции прибытия;
- по времени в пути.

## 2. Архитектура приложения

Приложение построено в соответствии с шаблоном **Model-View-Controller (MVC)** с пассивной моделью.

- **Model** – доменная модель `TrainRecord`, репозиторий `TrainRepository` (и его реализация `SQLiteTrainRepository`), сервисы для поиска, пагинации, импорта/экспорта XML.
- **View** – набор окон и виджетов на базе PySide6 (Qt): `MainWindow`, `AddTrainDialog`, `SearchDialog`, `DeleteDialog`, `PaginationWidget`. Все представления только отображают данные и передают действия пользователя контроллеру.
- **Controller** – `MainController`, который обрабатывает сигналы от представлений, взаимодействует с моделью и обновляет представления.

Все компоненты слабо связаны: контроллер получает ссылки на представление, репозиторий и сервисы через конструктор (внедрение зависимостей).

## 3. Реализация модели (Model)

### 3.1. Класс `TrainRecord`

Класс представляет одну запись о поезде. Поля:
- `record_id` (int, опционально) – идентификатор в БД;
- `train_number` (str) – номер поезда;
- `departure_station` (str) – станция отправления;
- `arrival_station` (str) – станция прибытия;
- `departure_datetime` (datetime) – дата и время отправления;
- `arrival_datetime` (datetime) – дата и время прибытия;
- `travel_time` (timedelta) – вычисляемое поле: разница между arrival и departure.

При создании выполняется валидация:
- обязательные поля не пусты;
- станции отправления и прибытия различны;
- дата прибытия позже даты отправления.

Методы для сравнения и поиска: `matches_train_number`, `matches_departure_date`, `departure_time_in_range`, и т.д.

### 3.2. Критерии поиска

В модуле `search_criteria.py` определены классы для каждого типа поиска:
- `TrainNumberOrDateCriteria`
- `TimeRangeCriteria`
- `StationCriteria`
- `TravelTimeCriteria`

Каждый критерий содержит необходимые поля и метод `is_valid()`.

### 3.3. Репозиторий `TrainRepository`

Абстрактный класс с методами:
- `insert(record) -> TrainRecord`
- `insert_many(records)`
- `get_all() -> list[TrainRecord]`
- `replace_all(records)`
- `delete_by_ids(ids) -> int`
- `clear()`

Конкретная реализация – `SQLiteTrainRepository`, использующая SQLite. Таблица `trains` содержит все поля + `travel_minutes` для возможности сортировки, но при загрузке время в пути пересчитывается.

### 3.4. Сервисы

- **`PaginationService`** – выполняет постраничное разбиение списка записей. Возвращает объект `PaginationResult` с текущей страницей, общим числом страниц и т.д.
- **`TrainSearchService`** – реализует логику поиска по заданному критерию. Вызывает соответствующие методы `TrainRecord`.
- **`XmlExportService`** – сохраняет список записей в XML-файл с использованием DOM-парсера (`xml.dom.minidom`). Формат соответствует структуре записи.
- **`XmlImportService`** – загружает записи из XML-файла с использованием SAX-парсера (`xml.sax`). Обработчик `_TrainContentHandler` собирает данные и создаёт объекты `TrainRecord`.

## 4. Реализация представлений (View)

### 4.1. Главное окно `MainWindow`

- Содержит меню («Файл», «Записи») и панель инструментов, дублирующую основные команды (добавить, поиск, удалить, сохранить XML, загрузить XML, открыть БД, сохранить БД как).
- В центральной части – `QTabWidget` с двумя вкладками:
  - **Таблица** (`QTableWidget`) – отображает записи в виде таблицы с колонками согласно варианту.
  - **Дерево** (`QTreeWidget`) – альтернативное отображение: каждая запись – корневой узел, поля – листья.
- Снизу размещён `PaginationWidget` – пользовательский виджет для управления страницами (кнопки «первая», «предыдущая», «следующая», «последняя», выбор количества записей на странице, отображение текущей страницы и общего числа записей).

Главное окно испускает сигналы (`add_requested`, `search_requested`, и т.д.), которые обрабатываются контроллером.

### 4.2. Диалог добавления записи `AddTrainDialog`

- Поля ввода: номер поезда, станции отправления и прибытия, два `QDateTimeEdit` с календарём для дат/времени.
- При изменении дат автоматически вычисляется и отображается время в пути (метка `travel_time_label`).
- Валидация перед закрытием: если данные некорректны – выводится предупреждение, диалог не закрывается.
- При успехе возвращает объект `TrainRecord` через метод `get_record()`.

### 4.3. Диалог поиска `SearchDialog`

- Верхняя часть: комбобокс для выбора режима поиска (4 варианта). При переключении меняется панель с критериями (`QStackedWidget`).
- Каждая панель содержит соответствующие элементы:
  - *Номер/дата*: `QLineEdit` для номера, чекбокс для включения даты и `QDateEdit`.
  - *Диапазон времени*: две группы для отправления и прибытия, каждая с чекбоксом включения и двумя `QTimeEdit` (от, до).
  - *Станции*: два `QLineEdit`.
  - *Время в пути*: два `QSpinBox` (часы, минуты).
- Кнопка «Найти» формирует критерий и испускает сигнал `search_requested(criteria)`.
- Нижняя часть: таблица результатов поиска и собственный `PaginationWidget` для навигации по результатам.

### 4.4. Диалог удаления `DeleteDialog`

Аналогичен диалогу поиска, но не содержит таблицы результатов. Кнопка «Удалить записи» формирует критерий и испускает сигнал `delete_requested(criteria)`. После подтверждения пользователем контроллер выполняет удаление и сообщает результат.

### 4.5. Виджет пагинации `PaginationWidget`

Содержит кнопки навигации, комбобокс выбора размера страницы, информационные метки. Испускает сигналы:
- `first_requested`, `previous_requested`, `next_requested`, `last_requested`;
- `page_size_changed(int)`.

Используется как в главном окне, так и в диалоге поиска.

## 5. Реализация контроллера `MainController`

Контроллер связывает представления и модель. Основные обязанности:

- Инициализация: подключает сигналы главного окна, загружает первую страницу из БД.
- Обработка добавления записи: открыть диалог, получить запись, вставить в репозиторий, обновить главное окно.
- Обработка поиска: открыть (или показать) диалог поиска, по сигналу `search_requested` выполнить поиск через `TrainSearchService`, обновить таблицу результатов в диалоге.
- Обработка удаления: открыть диалог удаления, по сигналу `delete_requested` найти записи, запросить подтверждение, удалить через репозиторий, обновить главное окно и показать количество удалённых.
- Сохранение/загрузка XML: через сервисы экспорта/импорта.
- Работа с файлами БД: смена текущего файла SQLite через диалоги `QFileDialog`, обновление репозитория и обновление отображения.
- Управление пагинацией как в главном окне, так и в окне поиска: обработка сигналов от `PaginationWidget` и перерисовка соответствующей таблицы.

Контроллер хранит текущий номер страницы для главного окна и для поиска, а также список результатов поиска.

## 6. Работа с базой данных

В качестве СУБД используется SQLite. Файл базы данных может быть указан пользователем через меню «Открыть БД» или «Сохранить БД как». При первом запуске создаётся база `data/train_records.db` в каталоге приложения.

Таблица `trains` создаётся со следующей схемой:
```sql
CREATE TABLE IF NOT EXISTS trains (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    train_number TEXT NOT NULL,
    departure_station TEXT NOT NULL,
    arrival_station TEXT NOT NULL,
    departure_datetime TEXT NOT NULL,
    arrival_datetime TEXT NOT NULL,
    travel_minutes INTEGER NOT NULL
);
```

Поле `travel_minutes` сохраняется для возможной сортировки, но при загрузке время в пути пересчитывается из дат.

## 7. Работа с XML

Экспорт в XML выполняется с помощью DOM-парсера (`xml.dom.minidom`). Создаётся корневой элемент `<trains>`, для каждой записи – элемент `<train>` с атрибутом `id` и дочерними элементами для каждого поля.

Импорт выполняется с помощью SAX-парсера. Обработчик `_TrainContentHandler` накапливает данные и создаёт объекты `TrainRecord`. SAX выбран для экономии памяти при больших объёмах данных.

Формат файла:

```xml
<?xml version="1.0" encoding="utf-8"?>
<trains>
    <train id="1">
        <train_number>001А</train_number>
        <departure_station>Минск</departure_station>
        <arrival_station>Гомель</arrival_station>
        <departure_datetime>2025-03-20T08:30:00</departure_datetime>
        <arrival_datetime>2025-03-20T12:45:00</arrival_datetime>
        <travel_minutes>255</travel_minutes>
    </train>
    ...
</trains>
```
## 8. Пагинация и дерево

- **Пагинация** реализована как в главном окне, так и в окне поиска. Пользователь может выбрать количество записей на странице (5, 10, 20, 50) и перемещаться по страницам. Пагинация работает на стороне сервиса, который выделяет нужный срез из полного списка.
- **Дерево** строится динамически: для каждой записи создаётся корневой элемент с названием «Поезд № …», дочерние элементы – пары «Поле – значение». Все узлы разворачиваются для наглядности.

## 9. Сценарии демонстрации системы

Для демонстрации подготовлены три XML-файла, каждый содержит не менее 50 осмысленных записей о поездах (например, рейсы Минск–Брест, Минск–Гомель, международные направления). Имена файлов: `trains_sample1.xml`, `trains_sample2.xml`, `trains_sample3.xml`.

### Сценарий 1: Загрузка данных из XML и просмотр

1. Запустить приложение.
2. Выбрать пункт меню «Файл» → «Загрузить XML».
3. Выбрать файл `trains_sample1.xml`.
4. После загрузки в главном окне отобразится таблица с записями. Проверить, что время в пути вычислено правильно.
5. Переключиться на вкладку «Дерево» – увидеть иерархическое представление.
6. Использовать элементы пагинации: перейти на следующую страницу, изменить число записей на странице.

### Сценарий 2: Поиск записей

1. Нажать кнопку «Поиск» на панели инструментов или выбрать пункт меню.
2. Выбрать режим «По номеру поезда или дате отправления», ввести номер (например, «002Б») и нажать «Найти».
3. В таблице результатов отобразятся найденные записи; пагинация также работает.
4. Повторить поиск в других режимах:
   - «По диапазону времени отправления»: включить диапазон, задать интервал, убедиться, что найдены поезда, отправляющиеся в это время.
   - «По станции отправления или прибытия»: ввести «Минск» в поле станции отправления.
   - «По времени в пути»: задать 2 часа 30 минут – найти поезда с таким точным временем.

### Сценарий 3: Добавление записи

1. Нажать «Добавить запись».
2. Заполнить поля (номер, станции, даты). При изменении дат автоматически обновится время в пути.
3. Нажать ОК. Запись должна появиться в главном окне (возможно, на последней странице, если размер страницы мал).
4. Проверить, что запись сохранилась в базе данных (перезапустить приложение с той же БД – запись должна быть).

### Сценарий 4: Удаление записей

1. Открыть диалог удаления.
2. Выбрать режим, например «По станции отправления или прибытия», ввести станцию «Гомель».
3. Нажать «Удалить записи». Появится окно подтверждения с количеством записей, которые будут удалены.
4. Подтвердить удаление. После выполнения главное окно обновится, записи исчезнут.
5. Если записей, соответствующих критерию, нет, появится сообщение «Подходящие записи не найдены».

### Сценарий 5: Сохранение в XML

1. Выбрать «Файл» → «Сохранить XML», указать имя файла.
2. Программа сохранит все записи из текущей базы в XML-файл.
3. Открыть файл в текстовом редакторе – структура соответствует ожидаемой.

### Сценарий 6: Смена файла базы данных

1. Выбрать «Файл» → «Открыть БД», указать другой `.db` файл (например, ранее сохранённый).
2. Главное окно перезагрузит данные из новой базы; в строке состояния отобразится путь к файлу.
3. Выбрать «Файл» → «Сохранить БД как», создать новую копию текущей базы.

### Сценарий 7: Работа с несколькими файлами

Загрузить поочерёдно три подготовленных XML-файла, каждый раз проверяя, что данные отображаются корректно и количество записей не менее 50. Убедиться, что пагинация правильно считает страницы.

- графический интерфейс с главным окном и диалогами;
- меню и панель инструментов;
- хранение в базе данных SQLite;
- импорт/экспорт XML с использованием DOM (запись) и SAX (чтение);
- постраничный вывод с настраиваемым размером страницы;
- альтернативное отображение в виде дерева;
- поиск и удаление по четырём условиям варианта;
- уведомление о количестве удалённых записей;
- возможность работы с несколькими файлами БД и XML.

Приложение построено с соблюдением принципов MVC, что обеспечивает лёгкость расширения и поддержки. Код документирован, использует корректные типы данных (datetime, timedelta), предусмотрена валидация ввода.
